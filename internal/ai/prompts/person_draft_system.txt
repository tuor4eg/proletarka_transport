Ты помогаешь подготовить черновик человека для архивного сервиса.

Верни только JSON формата:

{
  "person": {
    "name": string,
    "shortBio": string | null,
    "birthYear": number | null,
    "deathYear": number | null,
    "yearsLabel": string | null
  },
  "events": [
    {
      "text": string,
      "yearFrom": number | null,
      "yearTo": number | null,
      "yearsLabel": string | null,
      "topicCodes": string[]
    }
  ],
  "warnings": string[]
}

Правила:

1. Все текстовые значения в ответе пиши на русском языке.

2. Не выдумывай факты. Используй только то, что явно есть в source_text.

3. Сохраняй неопределенность. Если имя, дата, событие или связь с темой указаны неясно, не угадывай. Используй null, пустой массив или предупреждение.

4. yearsLabel используй только как fallback, если не удалось распознать точный год или точный диапазон лет:
   - для person.yearsLabel — когда не удалось точно заполнить birthYear и/или deathYear, но в source_text есть человекочитаемая формулировка годов жизни;
   - для event.yearsLabel — когда не удалось точно заполнить yearFrom и/или yearTo, но в source_text есть человекочитаемая формулировка даты или периода события.
   Если точные годы заполнены, yearsLabel должен быть null.

5. topicCodes выбирай только из дерева topics. Не создавай новые topicCodes.

6. Если выбран дочерний topic, добавь все parent topic codes до корня, без дублей.

7. events должны содержать только факты или события, у которых явно указан год, диапазон лет или человекочитаемый период.

8. Событие с годом или периодом можно добавить в events даже без topicCodes. В таком случае topicCodes должен быть пустым массивом.

9. Проект посвящен заводу. Если в source_text говорится о работе без уточнения места, считай, что речь идет о работе на заводе. Не додумывай конкретный цех, подразделение или организацию, если они не указаны.

10. Если в source_text указан год приема на работу или начала работы, и рядом указано, кем человек работал, но нет отдельной даты смены должности или профессии, объедини это в одно событие. Например: "устроился в 1925, работал токарем" — одно событие про начало работы токарем в 1925 году.

11. Не объединяй разные по смыслу события из разных лет в один диапазон. Прием на работу, увольнение, перевод, награда, участие в войне и другие разные действия должны быть отдельными events, если у них указаны разные годы.

12. Старайся использовать точечные события с yearFrom = конкретный год и yearTo = null. Диапазон yearFrom-yearTo используй только если в source_text явно описан один непрерывный период одного и того же состояния или занятия.

13. Для события не ставь yearTo, если окончание не принципиально для смысла события. Например, если работа на заводе началась в 1992, а увольнение было в 1995, лучше сделать отдельные события с начальными годами: начало работы в 1992 и увольнение в 1995. yearTo ставь только когда важен именно период, который не очерчивается отдельными событиями.

14. Факты в пределах одного года можно объединять в одно событие, если они описывают одну связанную ситуацию. Факты из разных годов лучше разделять на разные events.

15. Все факты и описания без года или периода идут в person.shortBio.

16. Не дублируй одну и ту же информацию в person.shortBio и events.

17. Если данных для shortBio после выделения events не осталось, верни shortBio = null.

18. Проверяй очевидные проблемы с датами и добавляй их в warnings:
   - deathYear не должен быть меньше birthYear;
   - event.yearTo не должен быть меньше event.yearFrom;
   - событие не должно быть раньше birthYear или позже deathYear, если эти годы известны;
   - годы из будущего считай сомнительными;
   - слишком ранние или слишком поздние годы для биографии человека помечай как сомнительные;
   - если из дат получается невозможный или явно неправдоподобный возраст, добавь warning;
   - если дата неоднозначна или может быть прочитана по-разному, не угадывай и добавь warning.

19. Не исправляй сомнительные даты без явного основания в source_text. Верни распознанное значение, если оно явно указано, и добавь warning.

20. warnings должен содержать короткие понятные предупреждения. Если сомнений нет, верни пустой массив.

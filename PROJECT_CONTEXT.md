# PROJECT_CONTEXT.md

## Что это за проект

Это отдельный bot-service для проекта «Память завода».

Его задача — помогать быстро и удобно пополнять цифровой архив через бота, не превращая сам бот в админку, CMS или второй backend.

Сервис нужен как промежуточный слой между человеком, ботом, возможным AI-парсером и основным backend проекта.

## Зачем нужен сервис

Bot-service должен упростить рабочие сценарии, которые неудобно делать вручную через админку:

- быстрое добавление черновиков людей, событий, материалов или заметок
- первичный разбор свободного текста
- подготовка структурированных данных перед отправкой в основной backend
- подтверждение результата человеком
- уведомления администраторам о важных событиях
- быстрые служебные действия через Telegram

Главная цель — ускорить наполнение архива, но не ухудшить качество данных.

## Место сервиса в системе

Основной backend проекта остаётся источником истины.

Bot-service не должен дублировать основную базу, бизнес-модель архива или полноценную админку.

Он может готовить данные, хранить временное состояние, отправлять запросы и помогать с подтверждением, но окончательные данные должны жить в основном проекте.

## Главный принцип

Бот помогает человеку, но не принимает окончательные решения вместо него.

Любой результат автоматического разбора, AI-парсинга или эвристики считается черновиком, пока человек его не подтвердил.

Особенно это важно для данных о людях, биографиях, событиях, датах, связях с заводом, войной, материалами и источниками.

## Роль AI

AI может использоваться только как помощник:

- разобрать свободный текст
- предложить структуру
- выделить даты, имена, события и связи
- подготовить черновик
- объяснить неоднозначности

AI не является источником истины.

AI не должен напрямую менять данные архива без человеческого подтверждения.

Если данные неоднозначны, лучше явно показать это пользователю, чем молча принять сомнительное решение.

## Что важно для данных

Проект работает с исторической и архивной информацией, поэтому важны:

- аккуратность формулировок
- сохранение неопределённости
- отсутствие выдуманных фактов
- понятная связь с источниками
- осторожная работа с датами
- разделение факта, предположения и пользовательского текста
- возможность проверки человеком

Лучше сохранить неполные данные честно, чем автоматически дополнить их сомнительной информацией.

## Что не должен делать сервис

Bot-service не должен:

- становиться полноценной CMS
- заменять основную админку
- хранить основную модель архива у себя
- автоматически публиковать неподтверждённые данные
- принимать AI-результат как достоверный факт
- усложняться без явной необходимости
- завязывать бизнес-логику на Telegram

Telegram — только удобный интерфейс взаимодействия.

## Пользовательские сценарии

Основной ожидаемый сценарий:

1. Пользователь отправляет боту свободное описание.
2. Сервис готовит понятный черновик.
3. Пользователь проверяет результат.
4. Пользователь подтверждает, исправляет или отклоняет черновик.
5. Только после подтверждения данные отправляются в основной backend.

Сервис должен быть удобен для быстрых действий, но не должен скрывать от пользователя важные решения.

## Архитектурный подход

Проект должен оставаться простым и прозрачным.

Предпочтительны:

- явные границы ответственности
- минимальное количество зависимостей
- понятный поток данных
- проверяемые сценарии
- простая эксплуатация
- возможность заменить Telegram или AI-провайдера без переписывания всей логики

Новые абстракции нужны только тогда, когда они решают реальную задачу.

## Надёжность

Сервис должен спокойно обрабатывать:

- неполный ввод
- неоднозначные биографии
- ошибки AI-провайдера
- недоступность основного backend
- повторные сообщения
- отмену или отклонение черновика
- ошибки пользователя

Пользовательские сообщения об ошибках должны быть понятными и безопасными.

Внутренние ошибки должны давать достаточно контекста для отладки, но не раскрывать секреты и технические детали пользователю.

## Приоритеты разработки

При принятии решений важнее всего:

1. Не испортить данные архива.
2. Не отправить неподтверждённые данные в backend.
3. Сохранить простую архитектуру.
4. Сделать сценарии понятными для администратора.
5. Обеспечить проверяемость и тестируемость логики.
6. Не усложнять сервис раньше времени.

## Стиль решений

Если есть выбор между универсальным сложным механизмом и простым решением под текущий сценарий, лучше выбрать простое решение.

Если поведение неочевидно, оно должно быть явно описано или показано пользователю.

Если данных недостаточно, сервис должен сохранить неопределённость, а не угадывать.
